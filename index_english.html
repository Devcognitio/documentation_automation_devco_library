Introduction

The purpose of this library is to improve your automation script if you are using SerenityBDD framework and Screenplay as a pattern. You can read more about this technologies here:

  • SerenityBDD
  • Screenplay Pattern

Motivation

¿Who has not dreamed of having superpowers?

We know that we don´t have the possibility to give you extra force, the ability to fly or get invisible but something that We can do is give you superpowers to your automation code.
Yeah, that's right. How many times have you noticed that every time you create a web or mobile automation project, you end up developing new functions for your actors? Serenity Screenplay is a powerful tool that allows you to develop highly readable and reusable tests. But sometimes, we need a little more abilities or performable actions to make our suite much more adjusted to the user's journey through the application. At Devco we are superpower makers, so we've developed a library that is well used with Screenplay, will make any user invincible on this journey. And you can also use it in single/cross-platform test suites. Awesome huh? So enough business talk. Let's get our hands dirty.

2. Getting Started

Our library is published on Maven Repository. Here is the official link

Maven Repository - Automation Devco Library

Setup

This library is primarily used as a Java dependency. We typically use a build tool (such as Maven or Gradle) to resolve the Automation Devco Library dependency.
If you are using Gradle it can be done as follows 

                
                    dependencies {
                        implementation group: 'co.com.devco.automation', name: 'automationDevcoLibrary', version: '2.0.2'
                    }

In case that you are using Maven: 

                
                  <dependency>
                    <groupId>co.com.devco.automation </groupId>
                    <artifactId>automationDevcoLibrary </artifactId>
                    <version>2.0.2 </version>
                  </dependency>

Locator Strategy

This utility allows you to handle cross-platform mobile automation (Android and iOS) using the same script. The advantage of using the Locator strategy is that you can use static and dynamic locators easily.

Available Methods for Locator Strategy

Android iOS Type
1 withAndroidXpathDynamic() withIosXpathDynamic() Dynamic Strategy
2 withAndroidXpathStatic() withIosXpathStatic() Static Strategy
3 withAndroidId() withIosId() Static Strategy
4 withAndroidName() withIosName() Static Strategy
5 withAndroidClassName() withIosClassName() Static Strategy
6 withAndroidAccesibilityId() withIosAccesibilityId() Static Strategy
7 withIosClassChain() Static Strategy
8 withIosPredicatedString() Static Strategy

Before to start with any strategy the first thing that you need to do is create 2 folders in your User interface layer. Pages and Locator

Dynamic Locator Strategy

In your locator folder, you will create a class and one by one the Locators objects that you need as the example.  Use the word Locator in the class name as a recommendation.


              public class LoginLocator {
                  public static final Locator TXT_USERNAME_LOCATOR =
                          locator()
                                  .withAndroidXpathDynamic("//android.widget.EditText[contains(@resource-id, '{0}')]")
                                  .withIosXpathDynamic("(//XCUIElementTypeStaticText[@name='{0}'])[1]");
              }

Create in your page folder a class with the same name that the class Locator created but in this case, it should have "Page" word in the name to differentiate it. In this class you will create your Target objects sending the Locator as a parameter. Important the methods to call the dynamic locator are locatedBy and send as a parameter dynamicElement() like this:


              public class LoginPage {
                  public static final Target TXT_USERNAME =
                      Target.the("Input to username").locatedBy(dynamicElement(TXT_USERNAME_LOCATOR));
                }

Now your automation it´s working with dynamic locators using xpath with the parameter {0}....{n}. When you call the Target on an Interaction or Task Class you just need to use the method of() on Target, send the parameter and that the library will do the rest.

Static Locator Strategy

For static locators is the same configuration in your Locator class


  public class LoginLocator {
    public static final Locator TXT_USERNAME_LOCATOR =
           locator()
                   .withAndroidId("textInput_name")
                   .withIosPredicatedString("label == 'userName'");
}

Something important is that in your Page class the method that you need to call is located() and as a parameter call the method theElementBy() sending the Static Locator that you create


  public class LoginPage {
      public static final Target TXT_USERNAME =
          Target.the("Input to username").located(theElementBy(TXT_USERNAME_LOCATOR));
    }

To Consider

  • In your class Locator, you define all the elements that belong to the view that you are going to automate. It is not one class per locator.
  • You can´t define a Locator with and static strategy in one OS and dynamic in another because the logic of your Interaction or your Task change and it will fail in one of those.
  • To static strategies, when you create the locators objects, the Builder shows all the strategies that you can use for an element to Android and iOS.

Mobile

In this section, you will find Actions and Abilities available to Android and iOS. You can use this in a Screenplay pattern with Serenity BDD because it was constructed to facilitate the work on this pattern.

Actions

For a general definition, actions do all of the direct interaction with the Selenium API, and allow us to pick a more readable title for what we’re doing. A clear example of this is Click.on(), so the actions that you are going to see here are focused on specific interactions with the elements or screen for mobile.

Drag Element

Drag Element is an Interaction in which you can "literally" drag and specific elements on-screen in different ways. This Interaction is useful for swipes where you need to drag an element to eliminated, or see more options, etc.

How to use it?


              Drag.theElement(Target.the()).to(UP)
              Drag.theElement(Target.the()).to(DOWN)
              Drag.theElement(Target.the()).to(LEFT)
              Drag.theElement(Target.the()).to(RIGHT)

Scroll

Scroll Interaction is an excellent help to your automation where you need to move to found elements on screen that are on the bottom or even on the top. Yes, it works in both ways, in addition you can send as a parameter the maximum attempts that you are going to do to find the element, in case that you can´t find it continue with your tests needs

How to use it?

To do an Scroll to Bottom you can use it like this:


            Scroll.untilVisibleTarget(Target).toBottom().untilMaxAttempts(Integer)

To do an Scroll to Top you can use it like this:


          Scroll.untilVisibleTarget(Target).toTop().untilMaxAttempts(Integer)

Abilities

Ability is the able that an Actor has to do things to perform their assigned tasks. Ability is different from an Interaction because the ability is more technical, handling something on the device, working with context, etc.

Change Driver Context

When you are working with a web app or hybrid app, sometimes you need to change the context because the elements change and the way how you interact too, so if this is your case you need to switch between the Web and native views to WEBVIEW or NATIVE context.

How to use it?

Use the following ability to switch to WebView context:


              ChangeDriverContext.toWebview()

Use the following ability to switch to Native context:


              ChangeDriverContext.toNative()

Choose App

This ability allows choosing an application that is running in the background and work it if you need to. You just need to send the app name that is running in background and that´s it. Important if the app is not open this ability is not going to work.

How to use it?


              ChooseApp.fromBackground("App Name")

Run App in Background

This ability runs the current app in background for a short time and then returns to the app as the main app. That helps to simulate a refresh, specific behavior of your test scenario, or help some elements appear in the XML to interact with them.

How to use it?


              RunAppInBackground.modeForASecond()

Hide Keyboard

Sometimes the keyboard hides some elements that you need to interact with. This simple ability helps to hide the keyboard to see all the elements in the view.
It´s just for Android

How to use it?


                Hide.theKeyboard()

Web

In this section, you will find Actions and Abilities available to Web pages. You can use this in a Screenplay pattern with Serenity BDD because it was constructed to facilitate the work on this pattern.

Abilities

Ability is the able that an Actor has to do things to perform their assigned tasks. Ability is different from an Interaction because the ability is more technical, open a browser, open a tab, etc.

Change Tab

Some web pages open new tabs when an interaction does something with an element. If your automation script needs to continue the test with the new tab you need to change it but the driver keeps focus on the current page. This Ability help to change easily between tabs.

How to use it?


                ChangeToTab.withIndex(Integer)

Open New Tabs

This ability allows opening another tab if you need it for your test.

How to use it?


                OpenNewTab.onBrowser()

Waits

In this section, you will find an specific explicit wait.

Stand By

This explicit wait is good for waiting for an element that specifies the time that you need and uses the visibility as a condition to continue.

How to use it?


                StandBy.visibilityOf(Target).forNoMoreThan(Integer).seconds

Utilities

In this section, you will find some utilities to handle external files. These utilities are not constructed with the screenplay format because are to use in a different context, if you want to read a PDF in your test or hide information in CSV files, these utilities will help you to manipulate the information.

Read Data from CSV

In an automation script, we can manage the data with multiple strategies, that include external files as .xlsx or .csv. This utility reads a CSV file and extracts the data that you need using the name of one column as a filter.


As a recommendation, your file must be named with data_ as prefix and then the name, for example, data_login.csv and be located in a data folder in your test´s resources folder.

How to use it?


            DataCSV.getDataCsv(fileName, filter)

PDF Manager

This is an utility to read a PDF file from a URL in the browser or a specific path on your computer in addition you catch the content of the file to do what you need with the information.

How to use it?

Read a PDF from browser sending URL as a parameter 

            ReadPdf.readPdfFromBrowser()
Read a PDF from file system sending path as a parameter 

            ReadPdf.readPdfFromFile()
Get the information from PDF File

            ReadPdf.getPdfLines()
